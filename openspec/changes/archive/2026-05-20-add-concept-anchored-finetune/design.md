## Context

Today's fine-tune loss is `α · CE(corpus) + (1-α) · τ² · KL(host ‖ forged)` (with `α=1.0` default reducing to v0.3 pure-CE). Both terms minimise behavioural divergence from the host. Neither demands that the forged model's residual stream organise around named, interpretable concepts.

`econ-sae` Phases 5.1 → 6.2 demonstrated that **supervised auxiliary heads, attached to the substrate during training, are the single biggest lever for downstream SAE interpretability**. The Phase 6.2 recipe (dual head: pooled + per-channel; focal loss for class imbalance) lifted regime mAUC from 0.595 to 0.991 in one training run. Polygram compression of the supervised SAE then showed cluster count saturating at exactly the supervised concept count — direct evidence the substrate concentrated around the labelled concepts.

The forge fine-tune is the natural sae-forge surface to inject this. The forged model's residual stream *is* the polygram feature basis by construction. Attaching supervised heads off the residual stream and adding their loss to the fine-tune objective is a small, additive change.

**This proposal extends `TrainingConfig` and `run_finetune` with one supervised concept-anchoring term, defaulting off, structurally analogous to `add-host-distillation-finetune-loss`'s `distill_alpha` knob.**

## Goals / Non-Goals

**Goals:**
- Add a supervised concept-anchoring loss term to fine-tune, defaulting off.
- Implement the Phase 6.2 dual-head + focal-loss recipe verbatim — pooled head over the residual stream, per-channel head over designated dims, focal-BCE for class imbalance.
- Define a `LabelSource` registry so future backends (corpus-tagged labels, host-probe labels) plug in without touching the loss code.
- Ship one backend: `polygram-clusters` pseudo-labels generated by running the pre-fine-tune forged model over a calibration corpus. Self-supervised; no external labels required.
- Preserve byte-identity with the current fine-tune path when the new fields are at their default values.

**Non-Goals:**
- Corpus-tag-based label sources (spaCy NER, POS, sentiment). Out of scope; queued as `add-concept-tag-corpus-preprocessing` follow-up.
- LLM-based safety / human-interest taggers. Same follow-up.
- Cross-feature attention block (the structural follow-up — econ-sae's `AttnWorldModel` analog). Substantial new code; deferred.
- Re-training the SAE itself. sae-forge consumes polygram-compressed SAEs.
- Automatic hyperparameter sweep for γ / β / α. Defaults reflect Phase 6.2; manual tuning.
- New row schema fields. Concept-anchoring provenance lives in fine-tune run metadata, not `ParetoFrontierRow`.

## Decisions

### Decision 1 — One additive scalar (`concept_alpha`), defaulting to 0.0

The concept term is composed *on top of* whatever the existing loss is (CE alone, or CE + distillation KL): `L_total = (1 - concept_alpha) · L_existing + concept_alpha · L_concept`. With `concept_alpha = 0.0` the entire concept branch is skipped (no label-source instantiation, no head construction, no extra forward), preserving byte-identity.

**Why one scalar, not a vector of per-term knobs**: matches the established `distill_alpha` precedent and keeps the user surface minimal. The pool/channel sub-weights are exposed as `concept_pool_weight` / `concept_channel_weight` because Phase 6.2 explicitly used both and the ratio matters for which kinds of features each head handles; γ is exposed because focal loss is the documented Phase 6.2 unlock.

**Alternative considered**: replace `distill_alpha` with a unified `loss_weights: dict[str, float]` knob. Rejected — bigger surface, breaks existing call sites, no immediate benefit.

### Decision 2 — Dual head: pooled + per-channel, both default-on when `concept_alpha > 0`

Phase 5.1 (pooled head alone) got 4/6 regime features. Phase 5.2 (per-channel head alone) got a different 4/6. Phase 6.2 (both, with focal loss on pooled) got 6/6. The dual head is the documented unlock.

Pooled head reads `mean_pool(residual, T)` and predicts all concepts — catches *distributed* encodings where one concept is spread across many features. Per-channel head reads `residual[..., -n_concepts:]` (one designated dim per concept) — catches *localised* encodings where one concept lives on a small slice.

**Why the last n_concepts dims, specifically**: matches econ-sae Phase 5.2 verbatim ("reserve the last 6 dimensions of h1 as direct regime channels"). Deterministic, simple, no special configuration. The last dims are typically the least-used by the unembedding in residual-stream LMs, so reserving them as concept channels minimises capacity contention with the LM head.

**Alternative considered**: make per-channel head optional, default off. Rejected — Phase 6.2 evidence is that *both* heads are needed for the full effect, and a user who sets `concept_channel_weight = 0` already has the off path.

**Alternative considered**: let users specify which residual dims are concept channels via `concept_channel_indices: list[int]`. Out of scope for v1 (noted in docs as future knob); the default last-N reservation is sufficient for v1's purposes.

### Decision 3 — Focal loss with γ default 2.0

Phase 6.2 confirmed focal loss + `pos_weight = 1.0` outperforms plain BCE + `pos_weight = (1-prev)/prev`. The reason: rare classes need *gradient pressure*, but `pos_weight` over-pumps them and steals SAE-allocation capacity. Focal loss down-weights well-classified examples specifically, so class imbalance is handled smoothly without rebalancing-driven capacity theft.

γ=2.0 is the Phase 6.2 value; we adopt it as default. γ=0.0 reduces to plain BCE for users who want to experiment.

**Why not `pos_weight`**: empirically inferior in Phase 6.1 vs Phase 6.2 head-to-head. Documented in econ-sae's Phase 6.2 section.

### Decision 4 — Label sources via a registry; ship one backend; design for extension

The `LabelSource` protocol is small:

```python
class LabelSource(Protocol):
    def prepare(self, model: nn.Module, iterator: Iterable) -> int:
        """Run any one-time calibration. Returns n_concepts."""
        ...
    def labels_for_batch(self, batch) -> Tensor:
        """Returns multi-hot labels, shape [B, T, n_concepts], values in {0, 1}."""
        ...
```

The v1 backend is `PolygramClusterLabelSource`. `prepare` runs `calibration_batches` no-grad student forwards, projects hidden states into the polygram feature space (via the existing pseudoinverse), records per-cluster firing distributions, and freezes a firing threshold per cluster. `labels_for_batch` projects fresh hidden states and applies the frozen threshold.

**Why polygram-clusters first**:
- Self-supervised: no external labelled data required. Works on any forge target.
- Direct line to econ-sae Phase 7.2's finding that polygram cluster count is the right concept axis.
- The forged model's residual stream is *already* the polygram feature space; projection is one matmul.

**Why a registry**: corpus-tag-based and LLM-tagger-based label sources are obvious next steps (queued as separate proposals). The registry lets them land without touching the loss code. Custom user-defined label sources (e.g., a lab's domain-specific concept set) plug in via `@register_label_source(...)`.

**Alternative considered**: hard-code polygram-clusters as the only source. Rejected — locks out the corpus-tag backend without good reason; the registry interface is two methods and adds negligible code.

### Decision 5 — Calibration runs at fine-tune start, labels are frozen for the run

The polygram-cluster firing distributions are recorded once, at fine-tune start, from the *pre-fine-tune* forged model. The labels are then frozen for the duration of the fine-tune.

Alternative: re-compute labels at every step from the *current* (fine-tune-in-progress) student. Rejected — would leak the optimisation target (the labels would chase the student's drifting activations, defeating the supervision). Phase 6.2 worked because the labels were a stable target.

Alternative: re-compute labels every N steps. Rejected — adds a knob (`label_refresh_interval`) for marginal benefit; the "freeze at start" choice is the simpler default that matches Phase 6.2.

### Decision 6 — Provenance in run metadata, not row schema

Concept-anchoring is per-run config, not per-sweep diagnostic. The fine-tune run metadata (already written to disk via `TrainingConfig.to_metadata()`) carries `concept_alpha`, `concept_label_source`, `n_concepts` (from `LabelSource.prepare`), and the label-source's own state-snapshot dict. `ParetoFrontierRow` is untouched.

**Why not the row**: row schema is for sweep-diagnostic comparison across runs. Two rows differing in `concept_alpha` are two different fine-tune recipes, not two different sweep axes — they don't go on the same Pareto frontier without a controlled study. Keep the row clean.

**Alternative considered**: add a `concept_anchored: bool` flag to the row. Rejected — fine-tune run metadata is the right place; adding row fields creates an obligation to update sweep tooling for a knob most users won't touch.

### Decision 7 — `concept_alpha == 0.0` is the disable path; no separate disable flag

When `concept_alpha == 0.0`, the entire branch is skipped. No `concept_enabled: bool` flag; no `enable_concept_anchor=False` kwarg. This matches `distill_alpha = 1.0`'s precedent of "default value is the disable path."

**Why**: one knob is simpler than two. Users who want to disable concept anchoring set `concept_alpha = 0.0`; the implementation tests for this and skips.

## Risks / Trade-offs

- **Per-step cost when concept_alpha > 0.** Two extra linear projections (pooled + per-channel) plus one polygram projection per batch. Total overhead < 5% for GPT-2-scale forges; spec includes a microbenchmark check at the v0.5.x release gate.

- **Label noise from polygram-clusters backend.** Labels are inferred from the pre-fine-tune student; they're necessarily approximate. Mitigation: focal loss (γ=2) is robust to label noise; recommended `concept_alpha=0.1` keeps it as nudge.

- **Concept anchoring vs. distillation pull.** A high `concept_alpha` against a host that doesn't naturally encode the chosen concepts would degrade KL. Mitigation: defaults to 0.0; recommended `0.1` start; docs warn against `> 0.3` without per-host validation.

- **Reserving the last N residual dims for the per-channel head is structurally consequential.** Those dims become concept-supervised "slots" while the rest of the basis stays distillation-driven. Mitigation: the choice is deterministic and matches Phase 5.2; advanced users will be able to override via `concept_channel_indices` in a future revision (not v1).

- **Registry is a new public surface.** External code can register custom backends. Mitigation: the protocol is small (two methods); semver applies to the protocol shape, not to the v1 backend's internals.

- **Backend selection mistakes.** Setting `concept_label_source` to an unknown key currently fails late (when `prepare` is called). Mitigation: `TrainingConfig.__post_init__` validates the key against `LABEL_SOURCE_REGISTRY.keys()` at construction time so misspellings fail fast.

- **The polygram-clusters backend requires the student model + the polygram pseudoinverse.** Both are available in the fine-tune context; tested in the smoke. For other backends, the protocol passes the model + iterator, which is enough for any reasonable label source.

- **No validation that the chosen concepts are actually meaningful.** A degenerate label source (e.g., a polygram dictionary with one giant cluster) would produce trivial labels and the supervision would do nothing useful. Mitigation: the polygram-clusters backend's `prepare` records `n_concepts`; if it's `<= 1`, `prepare` raises with a message naming the polygram report and suggesting a higher-rung encoding. Documented.
